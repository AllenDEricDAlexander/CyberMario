# Clocktower Agent Platform Design

**日期:** 2026-06-17

**目标:** 为 CyberMario 设计一个可真实开局的钟楼桌游 Agent 平台。平台以对局房间、魔典状态、规则引擎和 Agent 编排为核心，支持人类说书人、人类玩家、Agent 玩家、Agent 自玩和实时多人在线游玩。

**当前背景:** CyberMario 已有 Spring Boot WebFlux 后端、React + Ant Design 前端、ReactAgent 运行时、Agent Memory、RAG、MCP、审计和 RBAC 基础能力。本设计不另建独立模拟器，而是在现有 Agent 和管理后台体系内增加钟楼对局域模型、规则数据、房间事件流、魔典控制台和玩家视角。

---

## 1. 已确认产品决策

用户确认选择方案 A: 先做“可运行核心局”再扩展。

第一版必须覆盖:

- 三个基础剧本: 《暗流涌动》《黯月初升》《梦殒春宵》。
- 强规则引擎: 尽量自动校验和结算，不只做文本建议。
- 双用户模式: 用户既可以作为说书人，也可以作为玩家。
- 实时多人: 支持真实玩家在线进入同一局。
- Agent 对局: Agent 席位复用同一套房间、事件、状态机和视角隔离机制。
- 配板器: 支持人数、剧本、难度、混乱度、阵营优势等参数。
- 规则数据: 允许将规则、角色、术语、相克规则结构化落库，并结合 RAG 做规则查询。
- 前端和 API: 管理/游戏页面和后端 API 都需要提供。
- 首版体验内置说书人工具箱: 魔典、规则查询、夜晚顺序提醒、状态标记。

## 2. 产品边界

第一版目标不是做一个“百科聊天机器人”，而是做一个能真实开局的钟楼对局平台。系统核心是“对局房间 + 魔典状态 + 规则引擎 + Agent 编排”。

用户可以创建房间，选择剧本和人数，由配板器生成身份配置；说书人在魔典中确认或手动调整配置后发放身份。进入游戏后，系统根据夜晚顺序、角色能力、玩家状态和公开事件推进流程。玩家可以在自己的视角发言、私聊、提名、投票、查看公开信息；说书人可以在魔典里查看全局真相、标记状态、处理规则裁定。

Agent 自玩不单独做一套模拟器，而是复用同一套房间、事件、状态机和视角隔离。区别只是玩家席位由 Agent 控制，Agent 只能看到该席位允许知道的信息。

## 3. 使用场景

### 3.1 人类说书人开局

说书人创建房间，选择剧本、人数和配板偏好。配板器生成角色配置和理由，说书人可以接受、重 roll 或手动调整。系统校验配置是否合法，并在魔典中展示玩家座位、角色、阵营、状态和需要用到的提示标记。

### 3.2 人类玩家参与

玩家加入房间，只能看到自己的身份、公开状态、公开事件、公开聊天、被允许的私聊和规则查询入口。玩家可以发言、私聊、提名、投票，死亡后保留发言能力并受死亡投票限制。

### 3.3 Agent 席位参与

房间中的任意玩家席位可以由 Agent 控制。Agent 通过受限视角获取输入，生成公开发言、私聊、夜晚选择、提名和投票。Agent 不读取魔典真相，也不能访问其他玩家的私密信息。

### 3.4 全 Agent 自玩

一个说书人 Agent 管理对局，多个玩家 Agent 分别控制席位。系统按相同规则推进对局，保存完整事件日志、模型调用、工具调用、裁定和回放数据。人类用户可以实时旁观、暂停、接管说书人或玩家席位。

## 4. 核心模块

| 模块 | 职责 |
|---|---|
| 规则数据中心 | 保存剧本、角色、角色类型、阵营、能力文本、夜晚顺序、设置修正、术语、相克规则和来源链接 |
| 配板器 | 按人数、剧本、难度、混乱度、阵营优势生成候选配置，并输出合法性校验和配板理由 |
| 对局房间 | 管理房间、席位、玩家连接、Agent 席位、房主、说书人和实时状态 |
| 魔典状态 | 保存座位顺序、角色、阵营、生死、醉酒、中毒、疯狂、提示标记、夜晚标记、旅行者和传奇角色 |
| 规则引擎 | 处理人数配置、设置阶段增减角色、阶段推进、夜晚顺序、提名投票、处决、死亡、状态变更和胜负检查 |
| 说书人控制台 | 展示魔典、夜晚提醒、规则裁定、状态标记、公开播报、私密告知和手动覆盖入口 |
| 玩家视角 | 展示玩家可见信息、发言、私聊、提名、投票、规则查询和当前阶段提示 |
| Agent 编排 | 为说书人 Agent 和玩家 Agent 构造受限上下文，驱动行动，并把输出写回房间事件流 |
| 规则查询 | 结合结构化规则数据和 RAG，回答规则、术语、角色能力、相克规则和说书人建议问题 |
| 回放与审计 | 保存配置、事件、发言、私聊、投票、夜晚动作、裁定、Agent 推理链路和模型调用摘要 |

## 5. 数据和规则边界

规则数据分两层:

1. 结构化规则数据: 用于强规则引擎和配置校验，包括剧本、角色、类型数量、夜晚顺序、设置修正、状态枚举、相克规则索引。
2. 文档知识数据: 用于规则查询和说书人解释，包括规则概要、重要细节、术语、说书人建议、角色详解和相克说明。

强规则引擎优先处理确定性规则。对于涉及说书人裁量、角色信息误导、醉酒/中毒错误信息、平衡建议和复杂相克判例的内容，系统给出建议并允许说书人确认或覆盖。所有手动覆盖都记录为裁定事件。

首版规则覆盖边界:

- 完整支持建局、配板、身份发放、阶段推进、夜晚提醒、白天讨论、提名、投票、处决、死亡投票限制和基础胜负检查。
- 三剧本角色都录入并可用于配置、查询和魔典展示。
- 高复杂度角色能力先按“规则引擎可自动结算 + 说书人确认”模式分级实现。
- 相克规则先用于配置风险提示和规则查询，再逐步转为自动结算约束。

## 6. 视角隔离

视角隔离是平台的安全边界。

| 视角 | 可见信息 |
|---|---|
| 说书人 | 完整魔典、所有身份、所有状态、所有夜晚动作、所有私密信息、规则引擎建议 |
| 当前玩家 | 自己身份、自己收到的信息、公开事件、公开聊天、自己参与的私聊、自己的可用行动 |
| 旁观用户 | 由房间设置决定，默认只能看公开事件；回放可选择隐藏真相或完整复盘 |
| 玩家 Agent | 与对应玩家完全相同，不可访问魔典和其他玩家私密上下文 |
| 说书人 Agent | 完整魔典和规则工具，但输出到玩家前必须经过事件类型和可见性控制 |

所有事件都带 `visibility` 和 `recipient`。前端、API 和 Agent prompt 构造都必须从同一套可见性过滤器读取数据，避免“页面看不到但 Agent 看得到”的信息泄露。

## 7. 主要数据流

### 7.1 创建对局

1. 用户选择剧本、人数、玩家名称、是否启用 Agent 席位和配板偏好。
2. 配板器生成候选配置。
3. 规则引擎校验角色数量、设置修正、相克提示和人数要求。
4. 说书人确认配置。
5. 系统创建房间、席位、魔典初始状态和身份分发事件。

### 7.2 夜晚

1. 系统根据当前阶段和剧本夜晚顺序生成说书人夜晚 checklist。
2. 说书人控制台展示需要唤醒的角色、可跳过原因、建议提示标记和可执行动作。
3. 人类玩家或 Agent 玩家提交夜晚选择。
4. 规则引擎自动结算确定性效果；需要裁量时生成说书人确认项。
5. 黎明时生成公开死亡/复活播报，不公开原因。

### 7.3 白天

1. 房间进入公开讨论，允许发言和受控私聊。
2. 说书人或系统开启提名阶段。
3. 玩家发起提名，被提名者辩护，玩家投票。
4. 规则引擎按存活人数半数门槛、当日最高票和平票规则计算“即将被处决”状态。
5. 结束提名后执行处决，更新死亡、死亡投票标记、能力失效和阶段推进。

### 7.4 Agent 自玩

1. 房间事件流为每个 Agent 构造独立可见上下文。
2. 玩家 Agent 输出行动意图，系统按当前阶段验证是否合法。
3. 说书人 Agent 只通过说书人动作 API 操作魔典和播报。
4. 所有 Agent 行动写入同一套对局事件，供实时页面和回放复用。

## 8. 前端页面

第一版建议新增四个入口:

| 页面 | 目标 |
|---|---|
| 配板器 | 选择剧本、人数和偏好，生成配置，查看合法性、阵营平衡、混乱度和说书人说明 |
| 对局房间 | 玩家实时参与页面，包含公开聊天、私聊、身份信息、阶段提示、提名和投票 |
| 说书人魔典 | 说书人专用页面，包含座位环、身份、状态标记、提示标记、夜晚顺序、裁定和播报 |
| 规则查询 | 面向说书人和玩家的规则问答，按当前视角过滤不应泄露的上下文 |

页面风格应贴近现有后台系统，优先做高密度、可扫描、操作明确的工具界面，不做营销式落地页。

## 9. 后端 API 和实时通道

后端按领域拆分 Controller 和 Service:

- `ClocktowerScriptController`: 剧本、角色、规则数据查询。
- `ClocktowerBoardController`: 配板生成、校验、保存候选配置。
- `ClocktowerGameController`: 创建房间、加入房间、开始对局、阶段推进、结束对局。
- `ClocktowerGrimoireController`: 魔典读取、状态标记、说书人裁定、夜晚动作确认。
- `ClocktowerPlayerActionController`: 发言、私聊、夜晚选择、提名、投票。
- `ClocktowerRuleQueryController`: 规则查询和来源引用。
- `ClocktowerReplayController`: 事件日志、回放、审计查询。

实时通道优先复用项目 WebFlux 能力，使用 SSE 或 WebSocket 向房间成员推送可见事件。每个推送事件必须经过视角过滤。

## 10. 持久化设计

后续涉及数据库变更时，必须按项目规则为单次数据库变更新增一个 Flyway 版本文件，不修改既有 migration。设计层面建议数据表分为:

- 规则静态数据: script、role、night_order、setup_rule、jinx_rule、term、source_document。
- 对局运行数据: game_room、game_seat、game_state、grimoire_entry、status_marker、game_event、private_message。
- 配板数据: board_config、board_role、board_score、board_validation_issue。
- Agent 数据: agent_seat_config、agent_action_log、agent_visibility_snapshot。
- 回放审计: replay_event、rules_ruling、model_audit 关联。

规则静态数据可以先由 migration 或启动初始化导入基础数据。百科原文不直接作为硬编码提示散落在代码里，应以结构化数据和文档知识库两种事实源维护。

## 11. RBAC 和安全

需要接入现有 RBAC resource provider 和 preset role:

- 普通玩家可加入房间、查看自己的玩家视角、执行合法玩家动作。
- 说书人可创建房间、查看魔典、执行裁定、推进阶段。
- 管理员可维护规则数据、查看审计和管理异常对局。
- Agent 自玩和模型调用需要单独权限，避免普通用户误触高成本模拟。

所有对局 API 必须校验房间成员身份、席位归属和当前阶段。玩家动作必须由规则引擎再次校验，不信任前端按钮状态。

## 12. 测试与验证

实现计划阶段需要覆盖以下验证:

- 配板器单元测试: 人数、剧本、角色数量、设置修正、难度参数和非法配置。
- 规则引擎单元测试: 阶段推进、提名限制、投票门槛、平票、处决、死亡投票、基础胜负。
- 视角隔离测试: 玩家、说书人、Agent、旁观者对同一事件看到不同字段。
- API 权限测试: 非房间成员、错误席位、错误阶段、越权魔典访问。
- Agent 上下文测试: 玩家 Agent prompt 不包含魔典真相和其他玩家私密消息。
- 前端测试: 配板器、魔典状态标记、夜晚提醒、提名投票关键交互。
- 回归测试: 现有 Agent、RAG、MCP、Memory 和 RBAC 页面不受影响。

## 13. 分阶段交付建议

### Phase 1: 核心局可跑通

完成三剧本规则数据、配板器、房间、席位、身份发放、魔典、夜晚顺序提醒、状态标记、白天提名投票、处决和基础回放。Agent 席位先能参与公开发言、提名和投票。

### Phase 2: 角色能力自动化

按角色复杂度分层实现自动结算。先实现设置阶段、信息类、基础保护/中毒/死亡类角色，再实现高裁量和复杂交互角色。说书人确认项贯穿所有不确定能力。

### Phase 3: 实时多人和 Agent 混局强化

完善私聊、断线重连、旁观、接管 Agent、暂停恢复、Agent 策略配置、实时审计和完整回放。

### Phase 4: 规则知识和运营能力

完善规则 RAG、来源引用、判例库、相克规则自动化、规则数据维护页面、对局统计和配板推荐优化。

## 14. 明确不做

- 不把 Agent 自玩做成独立于真实房间的第二套系统。
- 不让玩家 Agent 读取魔典真相或其他玩家私密消息。
- 不依赖 LLM 自由发挥来决定硬规则是否合法。
- 不在第一版追求所有复杂角色互动完全自动化。
- 不自动启动项目或运行本地服务。
- 不修改既有 Flyway migration；后续数据库变更只新增 migration。

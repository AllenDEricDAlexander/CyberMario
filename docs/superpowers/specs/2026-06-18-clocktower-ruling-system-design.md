# Clocktower Ruling System Product Design

**日期:** 2026-06-18

**文档类型:** Product Doc

**目标:** 为 Clocktower 说书人裁定系统定义第一版产品能力。裁定系统要让说书人可以快速处理死亡、复活、处决、游戏结束、提名投票流程控制和误操作回退，同时支持真实状态与公开状态分离，覆盖假死等钟楼实战场景。

**背景:** 当前 Clocktower 模块已有房间、座位、魔典、夜晚顺序、待处理任务、玩家提名/投票基础数据和事件流。现有说书人动作入口过于轻量，死亡、复活、游戏结束、投票流程和裁定历史没有形成统一产品体验。标记功能只能作为魔典提醒，不能承担规则状态变更。第一版裁定系统应成为说书人的核心控台能力。

---

## 1. 产品问题

说书人在真实对局中需要处理大量“规则之外但必须落地”的判断:

- 夜晚死亡、白天处决、角色能力导致死亡或复活。
- 公开信息与真实魔典状态不一致，例如假死、公开伪装复活。
- 投票和处决流程中的误操作、重开、撤销和跳过。
- 游戏结束、胜方记录和结束原因。
- 裁定历史、公开广播和回放审计。

如果这些能力分散在标记、事件备注或临时按钮里，用户会遇到三个问题:

- 当前状态不可信: 标记写了“死亡”，但座位仍然存活。
- 玩家视角不一致: 公开页面不知道该展示死亡还是存活。
- 回放不可审计: 无法区分规则结果、说书人手动覆盖和误操作回退。

裁定系统的产品目标是把“说书人判断”变成一等对象: 可执行、可见、可撤销、可回放。

## 2. 用户和场景

### 2.1 人类说书人

说书人在魔典页处理所有裁定。他需要在局中快速点击常用动作，也需要在高风险动作前确认原因和影响范围。

典型场景:

- 黎明宣布 4 号死亡。
- 角色能力让 7 号复活。
- 公开显示 3 号死亡，但魔典真实状态仍为存活。
- 猎手能力导致一名邪恶玩家被处决。
- 当前提名投票误开，需要撤销并重开。
- 游戏结束，记录邪恶胜利和原因。

### 2.2 玩家

玩家只看到公开裁定结果，不看到魔典真实状态、内部原因和说书人备注。玩家看到的座位生死以公开状态为准。

### 2.3 回放查看者

回放需要看到裁定发生过什么、是否被撤销、公开事件如何影响玩家视角。完整回放可以展示裁定原因和说书人备注，公开回放只能展示公开结果。

## 3. 产品原则

- **裁定是状态变更，不只是备注。** 判死亡、复活、游戏结束等裁定必须修改当前对局状态。
- **裁定必须留痕。** 所有裁定写入裁定记录，并追加事件，不删除历史。
- **快操作和安全确认并存。** 常用死亡/复活可快捷执行，高风险动作必须确认。
- **真实状态和公开状态分离。** 魔典服务规则判断使用真实状态，玩家视角使用公开状态。
- **标记不是规则状态。** 标记保留为提醒贴纸，不再承担死亡、复活、游戏结束等核心规则含义。
- **先覆盖说书人手动控制，不自动推断全部角色规则。** 第一版不追求完整自动结算，优先保证人类说书人能控局。

## 4. 产品范围

### 4.1 第一版必须支持

- 独立裁定模块和裁定历史。
- 真实生死和公开生死两层状态。
- 判死亡、复活、公开生死裁定。
- 处决玩家、跳过处决。
- 结束游戏，胜方可选，原因必填。
- 阶段推进或指定阶段。
- 关闭当前提名、重开提名、撤销提名。
- 审计式撤销裁定。
- 魔典页座位快捷裁定和集中裁定面板。
- 玩家视角只显示公开状态。

### 4.2 第一版不做

- 不做每个玩家各自不同的一套生死状态。
- 不做完整角色能力自动结算。
- 不自动推断胜负，只接受说书人裁定。
- 不删除历史事件、投票或裁定记录。
- 不把标记升级成规则状态。
- 不实现复杂时间旅行式全局回放重算；撤销只恢复裁定记录保存的目标快照。

## 5. 核心概念

### 5.1 裁定

裁定是说书人创建的、有副作用的命令记录。创建裁定时系统会:

1. 校验说书人权限和请求字段。
2. 保存裁定记录。
3. 修改房间、座位或提名状态。
4. 追加公开或说书人可见事件。
5. 返回更新后的魔典和裁定结果。

### 5.2 真实状态

真实状态是魔典和规则判断使用的状态。例如一个玩家真实存活，就仍然可被规则视为存活，即使公开显示死亡。

### 5.3 公开状态

公开状态是玩家视角看到的状态。假死通过公开状态实现: 真实状态不变，公开状态显示死亡。

### 5.4 审计式撤销

撤销不删除原裁定。系统会把原裁定标记为已撤销，新增一条撤销裁定和撤销事件，并把当前状态恢复到原裁定保存的快照。

## 6. 裁定类型

| 裁定类型 | 用途 | 是否高风险 | 主要状态影响 |
|---|---|---:|---|
| `MARK_DEAD` | 判玩家死亡 | 否 | 默认同步真实死亡和公开死亡 |
| `RESTORE_ALIVE` | 复活玩家 | 否 | 默认同步真实存活和公开存活 |
| `SET_PUBLIC_LIFE` | 只修改公开生死 | 是 | 真实状态不变，用于假死或公开伪装 |
| `EXECUTE_PLAYER` | 处决玩家 | 是 | 记录处决，可导致真实和公开死亡 |
| `SKIP_EXECUTION` | 跳过当前处决 | 是 | 关闭处决结果，不改玩家生死 |
| `END_GAME` | 结束游戏 | 是 | 房间进入结束状态，记录胜方和原因 |
| `ADVANCE_PHASE` | 推进或指定阶段 | 否 | 修改当前阶段、天数或夜晚数 |
| `CLOSE_NOMINATION` | 关闭当前提名/投票 | 是 | 提名状态关闭，投票记录保留 |
| `REOPEN_NOMINATION` | 重开提名 | 是 | 指定提名重新打开 |
| `VOID_NOMINATION` | 撤销提名 | 是 | 指定提名作废，投票记录保留 |
| `UNDO_RULING` | 撤销裁定影响 | 是 | 恢复被撤销裁定保存的快照 |

## 7. 裁定原因

裁定原因使用固定原因加备注。固定原因用于筛选、回放和审计，备注用于说书人解释细节。

| 原因 | 含义 |
|---|---|
| `VOTE_EXECUTION` | 提名投票导致 |
| `ROLE_ABILITY` | 角色能力导致 |
| `NIGHT_DEATH` | 夜晚死亡结算 |
| `STORYTELLER_RULING` | 说书人裁量 |
| `PLAYER_REQUEST` | 玩家请求或玩家动作引起 |
| `MISTAKE_FIX` | 修正误操作 |
| `OTHER` | 其他原因 |

备注规则:

- 所有高风险裁定必须填写备注。
- `END_GAME` 必须填写原因备注。
- 公开备注可选；为空时系统生成简短公开文案。
- 内部备注只给说书人和完整回放可见。

## 8. 真实状态与公开状态

座位需要有两层生命状态:

- `lifeStatus`: 真实状态。
- `publicLifeStatus`: 公开状态。

产品规则:

- 魔典显示真实状态和公开状态。
- 玩家视角只显示公开状态。
- 普通死亡和复活默认同步修改两层状态。
- 假死只修改公开状态，不修改真实状态。
- 提名、投票和规则判断默认使用真实状态。
- 如果未来需要“公开死亡的人不能发言但真实仍存活”，另行设计公开交互状态，不在本版混入生命状态。

## 9. 前端体验

### 9.1 魔典座位快捷裁定

每个座位提供快捷裁定入口:

- 死亡。
- 复活。
- 公开生死。

死亡和复活可快速提交，但仍需要保留默认原因和可编辑备注入口。公开生死会打开确认表单，避免误造假死状态。

座位行需要展示:

- 真实状态。
- 公开状态。
- 死票是否可用。
- 最近相关裁定提示。

### 9.2 集中裁定面板

集中裁定面板处理高风险或非座位单点动作:

- 处决玩家。
- 跳过处决。
- 结束游戏。
- 指定阶段。
- 关闭、重开、撤销提名。
- 撤销裁定。

高风险动作使用确认表单。确认表单必须清晰展示:

- 将影响哪个玩家、提名或房间。
- 会修改哪些状态。
- 是否公开广播。
- 公开文案。
- 内部备注。

### 9.3 裁定历史

魔典页展示最近裁定历史:

- 裁定类型。
- 目标玩家或目标提名。
- 原因。
- 状态: 已应用、已撤销。
- 公开状态变化。
- 创建时间和说书人。
- 撤销入口。

被撤销裁定保留在历史中，不能消失。

## 10. API 产品契约

裁定使用独立 API，不复用通用说书人 action switch。

| API | 用途 | 权限 |
|---|---|---|
| `POST /api/clocktower/rooms/{roomId}/rulings` | 创建并应用裁定 | 说书人或管理员 |
| `GET /api/clocktower/rooms/{roomId}/rulings` | 查看裁定历史 | 说书人或管理员 |
| `POST /api/clocktower/rooms/{roomId}/rulings/{rulingId}/undo` | 撤销裁定 | 说书人或管理员 |

创建请求的产品字段:

| 字段 | 说明 |
|---|---|
| `rulingType` | 裁定类型 |
| `targetSeatId` | 目标座位，座位相关裁定必填 |
| `nominationId` | 目标提名，提名相关裁定必填 |
| `targetPhase` | 目标阶段，阶段裁定使用 |
| `publicLifeStatus` | 公开生死，公开状态裁定使用 |
| `winner` | 胜方，可为善良、邪恶或空 |
| `reason` | 固定原因 |
| `note` | 内部备注，高风险裁定和游戏结束必填 |
| `publicNote` | 公开文案，可选 |
| `visibility` | 公开或仅说书人 |

响应需要让前端无需二次猜测:

- 裁定记录。
- 更新后的魔典摘要。
- 关联事件列表。
- 被修改对象的前后状态摘要。
- 是否需要前端刷新提名、投票或房间状态。

## 11. 持久化产品要求

需要新增裁定主表，并为座位增加公开生命状态字段。数据库变更必须通过新的 Flyway migration 完成，不修改旧 migration。

### 11.1 裁定记录

裁定记录需要保存:

- 房间 id。
- 裁定类型。
- 裁定状态: 已应用、已撤销。
- 目标座位 id。
- 目标提名 id。
- 目标阶段。
- 胜方。
- 原因。
- 内部备注。
- 公开文案。
- 可见性。
- 被撤销的原裁定 id。
- 关联事件 id 列表。
- 创建人。
- 撤销人和撤销时间。
- 状态快照。

### 11.2 状态快照

状态快照用于撤销。第一版快照覆盖:

- 房间: 状态、阶段、当前天数、当前夜晚数。
- 座位: 真实生死、公开生死、死票可用状态。
- 提名: 状态、票数、是否已处决。

快照是裁定撤销的依据。它不用于任意时间点回放重算。

## 12. 事件和可见性

裁定生效后需要写事件:

- 死亡写 `PLAYER_DIED`。
- 复活写 `STORYTELLER_RULING`。
- 处决写 `PLAYER_EXECUTED`，如果导致死亡再写 `PLAYER_DIED`。
- 阶段变化写 `PHASE_CHANGED`。
- 游戏结束写 `GAME_ENDED`。
- 提名状态变化写 `STORYTELLER_RULING`。
- 撤销写 `STORYTELLER_RULING`，payload 标明撤销目标。

可见性规则:

- 默认状态变化公开。
- 内部原因和备注只给说书人。
- 公开事件使用 `publicNote` 或系统生成文案。
- 完整回放能看到裁定详情，公开回放只看到公开事件。

## 13. 流程控制规则

### 13.1 提名

- 关闭提名会把当前或指定提名状态改为关闭。
- 重开提名会把指定提名状态改回打开。
- 撤销提名会把指定提名标记为作废。
- 投票记录不删除。
- 存在后续相关裁定时，撤销提名需要确认。

### 13.2 处决

处决可以来源于当前提名，也可以由角色能力或说书人裁定指定任意玩家。

- 关联提名时，默认目标是被提名者。
- 无提名时，允许手动选择目标玩家。
- 处决原因必须选择固定原因。
- 处决不一定等同死亡，但第一版默认处决会导致死亡；如果不死亡，需要在确认表单中显式选择并填写原因。

### 13.3 游戏结束

- 游戏结束必须填写原因备注。
- 胜方可选。
- 游戏结束后房间进入结束状态。
- 游戏结束可以撤销，撤销后恢复结束前房间状态。

## 14. 撤销规则

撤销采用审计式:

1. 原裁定状态改为已撤销。
2. 创建 `UNDO_RULING` 裁定。
3. 恢复原裁定保存的快照。
4. 追加撤销事件。
5. 原事件、投票和裁定记录保留。

限制:

- 只能撤销裁定模块创建的裁定。
- 不能直接撤销玩家动作。
- 如果原裁定之后存在影响同一房间、座位或提名的后续裁定，前端必须展示风险确认。
- 强制撤销仍保留所有历史，避免回放断裂。

## 15. 权限和安全

- 只有房间说书人和管理员可以创建、查看和撤销裁定。
- 玩家不能看到内部备注、真实状态和裁定历史。
- 玩家视角座位生死只读取公开状态。
- Agent 玩家视角与普通玩家一致，不能读取裁定内部字段。
- 说书人 Agent 如果后续接入，必须通过同一裁定 API 操作状态。

## 16. 成功标准

第一版完成后，用户应能做到:

- 在魔典里一键判某玩家死亡或复活，并看到玩家视角同步变化。
- 创建假死裁定: 魔典真实存活，玩家视角公开死亡。
- 从当前提名进入处决，或手动处决任意玩家并记录原因。
- 关闭、重开和撤销提名，投票记录仍可回放。
- 结束游戏并记录可选胜方和必填原因。
- 撤销一次裁定，状态恢复，历史中能看到原裁定和撤销记录。
- 在公开回放中只看到公开事件，在完整回放中看到裁定详情。

## 17. 验收测试方向

后端验收:

- 判死亡和复活同步真实状态与公开状态。
- 公开生死裁定只修改公开状态。
- 游戏结束缺少原因时拒绝。
- 游戏结束胜方可以为空。
- 处决可关联提名，也可手动指定玩家。
- 提名关闭、重开、撤销不删除投票记录。
- 撤销裁定恢复快照并保留原事件。
- 非说书人不能创建或撤销裁定。

前端验收:

- 魔典座位显示真实状态和公开状态。
- 座位快捷裁定按钮可见。
- 高风险裁定显示确认表单。
- 游戏结束必须填写原因。
- 裁定历史显示已应用和已撤销。
- 撤销裁定后页面刷新为恢复后的状态。

迁移验收:

- 本次数据库变更新增且只新增一个 Flyway migration。
- 新字段 `public_life_status` 与旧 `life_status` 初始化一致。
- 既有 migration 不被修改。

## 18. 后续扩展

本版完成后，可以继续扩展:

- 每个玩家不同视角的公开状态。
- 角色能力自动生成建议裁定。
- 裁定模板和快捷键。
- 裁定风险提示，例如“公开死亡但真实存活会影响玩家可见行动”。
- 说书人 Agent 使用裁定 API 自动控局。
- 回放中按裁定还原公开视角和完整魔典视角。

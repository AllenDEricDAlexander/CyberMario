# Clocktower Game Nomination Vote Execution Design

**Date:** 2026-07-08

**Status:** User-approved approach A. This spec covers task 07 from `docs/clocktower_agent_tasks`: move nomination, vote, and daytime execution state into the new game-level model so HUMAN and AGENT players share one durable rule path. Implementation must be planned separately before code changes.

---

## 1. Background

Tasks 01 through 06 established the new Agent-capable Clocktower game path:

- `clocktower_game_seat` carries HUMAN and AGENT actors copied from room seats.
- `clocktower_game_public_mic_*` controls day speech order.
- `ClocktowerGameActionExecutor` accepts normalized HUMAN and AGENT action commands.
- The new game view reads from `clocktower_game_event`.

The old room action chain still writes nomination and vote state into old room/grimoire tables:

```text
/api/clocktower/rooms/{roomId}/actions
  -> ClocktowerActionServiceImpl
  -> ClocktowerNominationPo / ClocktowerVotePo
```

That path cannot represent Agent actors correctly and would split state from the new game view. Task 07 adds a game-level nomination, vote, and execution domain while preserving the old room path for compatibility until the later flow cutover task.

## 2. Goals

1. Add durable game-level nomination, vote, and execution tables.
2. Include the project audit fields on every new table: `created_by`, `updated_by`, `version`, and `deleted`.
3. Execute `NOMINATE` and `VOTE` through `ClocktowerGameActionExecutor`.
4. Allow HUMAN and AGENT seats to nominate, be nominated, vote, and spend a dead vote.
5. Block player/Agent nominations until the public mic session is closed when a current-day mic session exists.
6. Prevent duplicate daily nominations by nominator and duplicate daily nominations of the same nominee.
7. Prevent multiple open nominations for the same game.
8. Enforce one vote per nomination per game seat.
9. Close nominations through a game-level Storyteller endpoint and compute the day execution candidate.
10. Resolve execution through a game-level Storyteller endpoint.
11. Append `clocktower_game_event` records for opened nominations, votes, closed nominations, candidate updates, executions, deaths, and no-execution outcomes.
12. Leave old room action and flow services behavior unchanged.

## 3. Non-Goals

This task does not:

- Cut over `ClocktowerFlowServiceImpl` to the new game tables.
- Remove or modify old `ClocktowerNominationPo` / `ClocktowerVotePo` behavior.
- Implement night choices or night task resolution.
- Add frontend controls.
- Add Agent autonomous scheduling.
- Model script-specific nomination exceptions.
- Start the app or run browser verification.

## 4. Selected Approach

Use approach A: add a self-contained game-level nomination/vote/execution domain and let the task 06 executor delegate `NOMINATE` and `VOTE` into it.

Rejected alternatives:

- Rewriting the old room action/flow services now would combine task 07 with the later flow cutover and increase regression risk.
- Writing only game events without durable nomination/vote/execution rows would make close/resolve impossible to validate safely.
- Introducing a per-action Strategy hierarchy now is premature. The domain services isolate the new rules without adding an unnecessary handler tree.

Design pattern decision:

- Use Domain Service boundaries for `ClocktowerGameNominationService`, `ClocktowerGameVoteService`, and `ClocktowerGameExecutionService`.
- Keep `ClocktowerGameActionExecutor` as an application orchestrator that validates the actor and delegates action-specific domain rules.
- Do not introduce Strategy, Factory, or Chain of Responsibility for v1 because the variation points are limited and direct services remain clearer.

## 5. Package Boundary

Create one focused backend package:

```text
top.egon.mario.clocktower.game.nomination
  ├── dto
  ├── po
  ├── repository
  ├── service
  │   └── impl
  └── web
```

Responsibilities:

- PO classes map the three new tables.
- Repositories provide focused lookups and existence checks.
- Nomination service opens and closes nominations.
- Vote service records votes and dead-vote spending.
- Execution service computes and resolves the day execution candidate.
- Web controller exposes Storyteller-only close/resolve endpoints.
- Executor remains the HUMAN/AGENT action entry point for `NOMINATE` and `VOTE`.

### 6.1 `clocktower_game_nomination`

Columns:

- `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- `game_id BIGINT NOT NULL`
- `day_no INTEGER NOT NULL`
- `nominator_game_seat_id BIGINT NOT NULL`
- `nominee_game_seat_id BIGINT NOT NULL`
- `status VARCHAR(32) NOT NULL`
- `vote_count INTEGER NOT NULL DEFAULT 0`
- `required_votes INTEGER NOT NULL DEFAULT 0`
- `opened_at TIMESTAMP WITH TIME ZONE NOT NULL`
- `closed_at TIMESTAMP WITH TIME ZONE`
- `metadata_json TEXT`
- `created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP`
- `updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP`
- `created_by BIGINT`
- `updated_by BIGINT`
- `version BIGINT NOT NULL DEFAULT 0`
- `deleted BOOLEAN NOT NULL DEFAULT FALSE`

Indexes:

- `idx_clocktower_game_nomination_game_day` on `game_id, day_no`
- `idx_clocktower_game_nomination_nominator_day` on `game_id, day_no, nominator_game_seat_id`
- `idx_clocktower_game_nomination_nominee_day` on `game_id, day_no, nominee_game_seat_id`
- unique open nomination index on `game_id` when `status = 'OPEN' and deleted = false`

### 6.2 `clocktower_game_vote`

Columns:

- `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- `game_id BIGINT NOT NULL`
- `nomination_id BIGINT NOT NULL`
- `voter_game_seat_id BIGINT NOT NULL`
- `vote_value BOOLEAN NOT NULL`
- `used_dead_vote BOOLEAN NOT NULL DEFAULT FALSE`
- `status VARCHAR(32) NOT NULL`
- `metadata_json TEXT`
- audit columns matching `clocktower_game_nomination`

Indexes:

- unique index on `nomination_id, voter_game_seat_id` when `deleted = false`
- `idx_clocktower_game_vote_game_nomination` on `game_id, nomination_id`

### 6.3 `clocktower_game_execution`

Columns:

- `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- `game_id BIGINT NOT NULL`
- `day_no INTEGER NOT NULL`
- `nominee_game_seat_id BIGINT`
- `nomination_id BIGINT`
- `status VARCHAR(32) NOT NULL`
- `executed BOOLEAN NOT NULL DEFAULT FALSE`
- `resolved_at TIMESTAMP WITH TIME ZONE`
- `metadata_json TEXT`
- audit columns matching `clocktower_game_nomination`

Indexes:

- unique index on `game_id, day_no` when `deleted = false`
- `idx_clocktower_game_execution_game_status` on `game_id, status`

No foreign keys are added, matching the existing Clocktower migration style.

## 7. Domain Rules

### 7.1 Nominate

`NOMINATE` uses `GameActionCommand.targetGameSeatIds().getFirst()` as the nominee.

Rules:

1. Game must be `RUNNING`.
2. Game phase must be `DAY` or `NOMINATION`.
3. Actor seat must be ACTIVE and a HUMAN or AGENT player.
4. Nominee seat must be ACTIVE and in the same game.
5. Nominator and nominee must both have `life_status = ALIVE` and `public_life_status = ALIVE`.
6. Self-nomination is rejected.
7. If a current-day public mic session exists, it must be `CLOSED`.
8. There must be no open nomination for the game.
9. Nominator must not have nominated today.
10. Nominee must not have been nominated today.
11. `required_votes = (alivePlayerCount + 1) / 2`.
12. Successful nomination sets game phase to `NOMINATION` and appends `NOMINATION_OPENED`.

### 7.2 Vote

`VOTE` uses `GameActionCommand.nominationId()` when present, otherwise the current open nomination.

Rules:

1. Nomination must be `OPEN`.
2. Actor seat must be ACTIVE and a HUMAN or AGENT player.
3. `vote` is required and must be explicitly true or false.
4. The actor may vote only once per nomination.
5. Alive players may vote true or false.
6. Dead players may vote false without spending a dead vote.
7. Dead players may vote true only when `has_dead_vote = true`.
8. Dead true votes set `used_dead_vote = true` and update the game seat to `has_dead_vote = false`.
9. Vote count is recalculated from durable yes votes after save.
10. Successful vote appends `VOTE_CAST`.

### 7.3 Close Nomination

`POST /api/clocktower/games/{gameId}/nominations/{nominationId}/close`

Rules:

1. Caller must be the room owner/Storyteller for the game room.
2. Nomination must be open.
3. Close sets `closed_at`, `status = CLOSED`, and final `vote_count`.
4. Execution candidate is recomputed from all closed nominations for the same game/day.
5. A unique top nomination with `vote_count >= required_votes` becomes the candidate.
6. A tie for top qualifying votes clears the candidate for v1.
7. Close appends `NOMINATION_CLOSED`.
8. Candidate recomputation appends `EXECUTION_CANDIDATE_UPDATED`.
9. Close keeps the game phase in `NOMINATION`, allowing additional same-day nominations before the Storyteller resolves execution.

### 7.4 Resolve Execution

`POST /api/clocktower/games/{gameId}/executions/resolve`

Request:

```java
public record ClocktowerGameExecutionResolveRequest(
        Boolean execute,
        Long targetGameSeatId,
        Long nominationId,
        String note
) {
}
```

Rules:

1. Caller must be the room owner/Storyteller for the game room.
2. A day execution row must exist.
3. `execute = true` requires a current candidate and matching `targetGameSeatId` / `nominationId`.
4. `execute = false` is allowed only when no current candidate exists.
5. Executing updates the target seat to `life_status = DEAD` and `public_life_status = DEAD`.
6. Executing appends `PLAYER_EXECUTED` and `PLAYER_DIED`.
7. No execution appends `NO_EXECUTION`.
8. Resolve sets `status = RESOLVED`, `executed`, and `resolved_at`.
9. Resolve sets the game phase to `EXECUTION` before appending execution/no-execution events.
10. Resolve does not advance to night; task 08 owns full flow transition.

## 8. Events

All accepted state-changing actions append public `clocktower_game_event` rows:

- `NOMINATION_OPENED`
- `VOTE_CAST`
- `NOMINATION_CLOSED`
- `EXECUTION_CANDIDATE_UPDATED`
- `PLAYER_EXECUTED`
- `PLAYER_DIED`
- `NO_EXECUTION`

Payloads include durable ids where relevant:

- `nominationId`
- `nominatorGameSeatId`
- `nomineeGameSeatId`
- `voterGameSeatId`
- `voteValue`
- `usedDeadVote`
- `voteCount`
- `requiredVotes`
- `executionId`
- `candidateGameSeatId`

Rejected actions keep using `ACTION_REJECTED` through the executor.

## 9. API Surface

Existing task 06 endpoint gains real support for two action types:

```text
POST /api/clocktower/games/{gameId}/actions
```

New Storyteller endpoints:

```text
POST /api/clocktower/games/{gameId}/nominations/{nominationId}/close
POST /api/clocktower/games/{gameId}/executions/resolve
```

The new endpoints use the same reactive wrapper style as existing Clocktower controllers.

## 10. Validation Plan

Focused tests:

- Migration/JPA mapping coverage for the three new POs.
- Executor nomination success for HUMAN and AGENT seats.
- Nomination rejected when mic is still open.
- Nomination rejected for self, duplicate nominator, duplicate nominee, and existing open nomination.
- Vote success for alive HUMAN/AGENT seats.
- Dead false vote does not spend dead vote.
- Dead true vote spends dead vote.
- Duplicate vote is rejected.
- Closing a nomination creates/updates the execution candidate.
- Tied qualifying nominations clear the candidate.
- Resolving an execution kills the candidate and appends events.
- Resolving no-execution records `NO_EXECUTION`.

Verification commands:

```bash
./mvnw -pl be -Dtest=ClocktowerGameNominationServiceTests,ClocktowerGameActionServiceTests test
./mvnw -pl be test
```

## 11. Risks And Follow-Ups

- Task 08 must decide when to advance from `EXECUTION` to `NIGHT` or the next day.
- Script-specific nomination exceptions are not modeled in v1.
- The old room action and flow endpoints continue to use old tables until the planned cutover.
- Frontend controls may need richer available-action metadata after this backend slice lands.
